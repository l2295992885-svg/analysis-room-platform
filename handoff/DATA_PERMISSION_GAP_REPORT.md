## 1. 结论

每日 LKJ 已接入记录级数据权限第一版。相比阶段 13，当前不再只依赖前端按钮、菜单权限或请求参数过滤，后端服务层已经在查询、详情、流程动作、附件、导出、结果库、待办、信箱、聊天业务卡片打开等路径上叠加每日 LKJ 记录级范围校验。

本轮 `scripts/acceptance/daily-lkj-mvp-check.ps1` 已完成多角色端到端实跑，结果为 `PASS=56, FAIL=0, TODO=1`。多角色数据范围、待办打开、附件下载、导出、聊天卡片打开业务的核心断言均通过。

## 3. 当前角色范围规则

| 角色 | 当前第一版范围 |
| --- | --- |
| 分析员 | 本人创建/提交的记录 |
| 分析班长 | 同创建部门或责任部门范围内记录，作为“所属分析员”的最小实现 |
| 分析室主任 | 第一版按分析室主任角色可查看每日 LKJ 范围记录 |
| 车间 | 下发到本车间 ID 或名称匹配的记录 |
| 车队 | 下发到本车队 ID 或名称匹配的记录 |
| 指导组 | 下发到本指导组 ID 或名称匹配的记录 |
| 超级管理员 | 技术管理绕过数据范围；业务历史写操作仍写日志并受显式权限约束 |

## 5. 仍需补强的问题

| 缺口 | 说明 |
| --- | --- |
| 真实组织隶属 | 车间、车队、指导组应来自正式组织表，不应长期依赖名称兜底匹配 |
| 班长范围 | 应明确班长管理哪些分析员，不能长期只按部门近似 |
| 主任范围 | 应明确分析室主任的数据边界，是全分析室还是分管范围 |
| 超级管理员业务边界 | 当前超级管理员有技术绕过能力，生产需补“显式业务授权 + 强审计”策略 |
| SQL 性能 | 结果库数据范围通过可见记录 ID 过滤，数据量大后需要改为 join/subquery 或专用范围表 |
| 公共文件中心 | 当前每日 LKJ 附件已校验记录范围，公共文件中心跨业务下载仍需补二次授权 |
