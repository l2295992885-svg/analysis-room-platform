# 平台权限模型与每日 LKJ 音视频违标公示权限矩阵

本文定义“综合分析室数据分析平台”的权限模型，以及“违章管理 > 每日 LKJ 音视频违标公示”的第一阶段角色权限矩阵。本文是后续菜单、按钮、接口、数据权限、字段权限、导出权限和附件权限实现的业务约束。

每日 LKJ 音视频违标公示是“违章管理”模块下的样板业务，不是整个平台本身。权限设计必须平台化，后续 LKJ 分析、音视频分析、单兵分析、统计分析等模块应复用同一套权限模型。

## 1. 权限类型

| 权限类型 | 含义 | 后端校验要求 |
| --- | --- | --- |
| 功能权限 | 能否进入菜单、页面或功能入口 | 进入页面前后端都可判断，后端仍需校验接口权限 |
| 操作权限 | 能否执行新增、提交、审核、退回、下发、反馈、入库等动作 | 所有动作接口必须校验 |
| 数据权限 | 能查看和处理哪些组织、人员、批次、流程记录 | 所有查询和动作接口必须校验 |
| 字段权限 | 能否查看、编辑或授权覆盖关键字段 | 所有字段修改必须校验 |
| 导出权限 | 能否导出当前页面或当前业务数据 | 独立于查看权限，所有导出必须校验并写日志 |

权限原则：

1. 前端按钮隐藏只用于体验，不构成安全控制。
2. 后端必须根据当前用户、角色、部门、数据范围、当前状态和目标对象范围重新校验。
3. 能进入页面不代表能操作；能查看数据不代表能导出；能收到聊天卡片不代表有权限查看或处理。

## 2. 平台角色

| 角色 | 定位 | 权限边界 |
| --- | --- | --- |
| 分析员 | 负责录入、导入、预览、提交、处理退回修改 | 默认只能处理本人草稿、本人提交、退回给本人的数据 |
| 分析班长 | 负责审核所属分析员提交的数据 | 默认只能处理所属分析员提交的数据 |
| 分析室主任 | 负责主任审核、下发、复核、最终确认、入库或撤销 | 默认可看分析室范围数据、责任部门返回数据、最终确认数据 |
| 车间 | 接收主任下发数据，并下发车队 | 默认只能处理下发到本车间的数据 |
| 车队 | 接收车间下发数据，并下发指导组 | 默认只能处理下发到本车队的数据 |
| 指导组 | 确认无误或反馈不属实 | 默认只能处理本指导组待确认和已处理数据 |
| 系统管理员 | 管理用户、角色、部门、菜单、字典、权限和系统配置 | 管理配置不等于可以随意修改业务历史数据 |

4. 附件查看和下载必须独立授权；能查看业务记录不等于能下载附件。

## 5. 数据权限矩阵

| 角色 | 默认可见数据 | 默认可处理数据 | 不允许 |
| --- | --- | --- | --- |
| 分析员 | 本人草稿、本人提交、退回给本人的数据、本人待处理数据 | 本人草稿、退回给本人的记录、本人可提交批次 | 查看其他分析员草稿；处理未退回给本人的记录 |
| 分析班长 | 所属分析员提交的数据、班长待审核数据、本人已处理审核数据 | 所属分析员提交且处于班长审核范围的数据 | 审核非所属分析员数据；跨分析班组查看未授权数据 |
| 分析室主任 | 分析室范围数据、责任部门返回数据、最终确认数据、主任待审核和待复核数据 | 主任待审核、主任下发、返回复核、最终确认、入库或撤销数据 | 绕过流程直接修改历史记录 |
| 车间 | 下发到本车间的数据、本车间已处理数据 | 本车间待下发车队的数据 | 查看其他车间数据；下发到非本车间下属车队 |
| 车队 | 下发到本车队的数据、本车队已处理数据 | 本车队待下发指导组的数据 | 查看其他车队数据；下发到非本车队下属指导组 |
| 指导组 | 本指导组待确认和已处理数据 | 本指导组待确认记录 | 查看其他指导组数据；处理非本指导组记录 |
| 系统管理员 | 系统配置、用户、角色、部门、菜单、字典等配置数据 | 系统配置数据 | 随意修改业务历史数据；绕过业务流程代替业务角色处理 |

数据权限实现要求：

1. 所有列表、详情、附件、日志、导出接口都必须套用数据权限。
2. 前端传入的部门、车间、车队、指导组筛选条件只能作为查询条件，不能作为授权依据。
3. 后端应基于登录用户的角色、部门、组织层级、业务归属和待办归属计算数据范围。
4. 分享业务卡片、信箱消息、待办消息只提供入口，不扩大数据权限范围。

## 6. 字段权限矩阵

字段分类：

| 字段分类 | 字段示例 |
| --- | --- |
| 录入基础字段 | 提报日期、违章发生日期、时间、责任部门、工号、责任人、机车、车次、地点、拟考核内容 |
| 编码字段 | 违标编码、性质、类别、类型 |
| 流程字段 | 当前状态、当前节点、提交人、审核人、下发对象、处理意见、最终意见 |
| 反馈字段 | 不属实原因类型、说明、证据附件 |
| 结果库字段 | 入库快照、结果版本、入库时间、是否计入 |
| 日志字段 | 操作人、操作时间、操作前状态、操作后状态、字段变更明细 |

| 阶段 | 录入基础字段 | 编码字段 | 流程字段 | 反馈字段 | 结果库字段 | 附件 | 说明 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 草稿阶段 | 可编辑 | 可选择；性质/类别/类型由编码带出；不一致需授权覆盖 | 只读 | 不可见 | 不可见 | 可上传、可删除本人草稿附件 | 分析员可修改未提交记录 |
| 分析员已提交后 | 只读 | 只读 | 只读 | 不可见 | 不可见 | 按授权查看 | 关键字段锁定，必须退回或授权修改 |
| 审核阶段 | 只读；退回后可由分析员修改 | 只读；授权覆盖需特殊权限和原因 | 可写审核意见、退回原因 | 不可见 | 不可见 | 按授权查看 | 班长和主任只能写审核意见，不直接改业务事实字段 |
| 下发确认阶段 | 只读 | 只读 | 可写下发对象、下发意见、确认意见 | 指导组处理前不可见 | 不可见 | 按授权查看 | 车间、车队负责下发，指导组负责确认 |
| 不属实反馈阶段 | 只读 | 只读 | 可写反馈流转意见 | 可编辑原因类型、说明；证据附件可上传 | 不可见 | 指导组可上传证据附件；相关处理人按权限查看 | 反馈不能删除原记录 |
| 结果库阶段 | 只读 | 只读快照 | 只读 | 只读 | 只读；更正需新版本 | 只读或按授权下载 | 不允许覆盖历史，只能走更正版本 |

字段权限规则：

1. 工号输入后自动带出责任人；手动改责任人必须有字段权限并记录原因。
2. 违标编码必须来自数据库；性质、类别、类型由编码带出。
3. 编码、性质、类别、类型不一致时必须阻止提交；确需覆盖时必须校验授权覆盖权限并写字段变更日志。
4. 已提交后不得直接编辑关键字段；必须退回修改或走授权修改流程。
5. 已入结果库后不得覆盖历史字段，只能生成新版本或追加更正记录。

## 7. 导出权限独立规则

导出权限独立于查看权限：

1. 能看不等于能导出。
2. 所有导出必须由后端校验导出权限。
3. 所有导出必须写业务日志和导出记录。
4. 导出内容必须受数据权限约束。
5. 导出条件、导出人、导出时间、导出数量、文件名、导出页面必须记录。
6. 导出失败也应记录失败原因，便于审计。

每日 LKJ 音视频违标公示导出特殊规则：

| 规则 | 要求 |
| --- | --- |
| 标题/副标题 | 必须包含提报日期 |
| 表格日期列 | 只导出违章发生日期 |
| 提报日期列 | 不单独导出提报日期列 |
| 数据范围 | 必须按当前用户数据权限过滤 |
| 附件 | 默认不随 Excel 导出，除非后续另行授权并记录日志 |
| 敏感字段 | 工号、责任人等字段后续可按字段权限决定脱敏策略 |

## 8. 聊天业务卡片权限规则

聊天业务卡片是提醒和跳转入口，不是授权凭证。

| 场景 | 权限规则 |
| --- | --- |
| 收到业务卡片 | 不代表拥有查看权限 |
| 收到审核卡片 | 不代表拥有审核权限 |
| 收到下发卡片 | 不代表拥有下发权限 |
| 收到反馈卡片 | 不代表拥有反馈权限 |
| 点击业务卡片 | 必须重新校验功能权限、数据权限和流程状态 |
| 卡片关联附件 | 必须重新校验附件查看或下载权限 |
| 卡片被转发 | 不扩大任何业务数据权限 |

后端要求：

1. 打开卡片详情时必须校验 `violation:daily:view` 和数据权限。
2. 点击卡片上的审核、下发、反馈、确认按钮时，必须按对应动作重新校验操作权限、数据权限和当前状态。
3. 聊天卡片发送、业务分享、文件分享、文件下载都必须写日志。

## 9. 后端权限原则

统一校验顺序：

```text
登录态 → 功能权限 → 操作权限 → 数据权限 → 当前状态 → 字段权限 → 附件权限/导出权限 → 写业务日志
```

1. 前端按钮隐藏只用于体验，不能作为安全依据。
2. 所有查询接口必须校验数据权限。
3. 所有详情接口必须校验数据权限。
4. 所有动作接口必须校验操作权限、数据权限、当前状态。
5. 所有字段修改必须校验字段权限。
6. 所有附件查看、下载必须校验附件权限和业务数据权限。
7. 所有导出必须校验独立导出权限，并写导出日志。
8. 状态流转的目标状态必须由后端根据当前状态和动作计算，不能由前端直接传入。
9. 系统管理员不默认拥有修改历史业务数据的权力。
10. 任何绕过业务日志、字段变更日志、状态机校验的写入都不允许进入结果库。

## 11. 后续需要业务确认的问题

1. 系统管理员是否允许临时代办业务动作；如果允许，是否必须双人确认或额外审批。
2. 分析室主任是否允许代替分析班长审核，还是必须按班长节点完成。
3. 分析班长和分析员的归属关系如何维护：按部门、按配置表，还是按角色数据范围。
4. 车间、车队、指导组的组织层级是否完全来自部门树，还是需要单独业务关系表。
5. 哪些角色可以拥有 `violation:daily:override` 授权覆盖关键字段权限。
6. 附件查看和附件下载是否分离授权；证据附件是否比普通附件更严格。
7. 导出权限是否按页面区分：列表导出、预览导出、结果库导出、日志导出。
8. 聊天业务卡片是否允许跨部门发送；如果允许，接收人无权限时展示什么提示。
9. 结果库阶段是否允许主任发起更正流程；如果允许，需要补充更正权限码。
10. 是否需要字段级脱敏规则，例如普通角色查看工号、姓名、附件时是否脱敏。
