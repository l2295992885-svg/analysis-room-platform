# 每日 LKJ 音视频违标公示业务需求

本文定义“违章管理 > 每日 LKJ 音视频违标公示”的第一阶段样板业务需求。该业务用于沉淀平台的录入、Excel 导入、编码校验、表格预览、审核流转、信箱、待办、附件、日志证据链、导出和结果库能力。

## 1. 业务定位

每日 LKJ 音视频违标公示是“违章管理”模块下的样板业务，不是整个平台本身。

平台后续还会扩展 LKJ 分析、音视频分析、单兵分析、解锁统计、调度命令、施工作业、试运机车、月度报表、统计分析、基础数据、系统管理等模块。本业务只负责跑通第一阶段最重要的业务闭环，并作为后续业务模块复用公共能力的参考样板。

本业务必须复用平台公共底座：

- 用户、角色、部门、菜单、按钮权限。
- 功能权限、操作权限、数据权限、字段权限。
- 登录日志、操作日志、业务证据链日志。
- 信箱中心、待办中心、附件管理、Excel 导入导出。
- 违标编码字典、人员基础数据、组织层级关系。

## 2. 业务流程

完整流程如下：

```text
分析员录入 / Excel导入
→ 违标编码校验
→ 表格预览
→ 分析员提交
→ 班长审核
→ 分析室主任审核
→ 主任下发车间
→ 车间下发车队
→ 车队下发指导组
→ 指导组确认无误 / 反馈不属实
→ 逐级返回
→ 分析室复核
→ 主任最终确认
→ 进入 LKJ / 音视频违标结果数据库
```

流程控制原则：

1. 记录只能按允许的流程节点流转，不能跳过当前节点。
2. 每次提交、审核、下发、确认、反馈、复核、最终确认都必须写业务证据链日志。
3. 每次流转都必须更新业务状态，并生成对应的待办和信箱消息。
4. 指导组反馈不属实不能直接删除原违章记录，只能进入返回复核链路。
5. 主任最终确认后，记录才能进入结果库。
6. 没有完整日志证据链的数据不得进入结果库。

## 3. 角色

| 角色 | 主要职责 |
| --- | --- |
| 分析员 | 手工录入、Excel 导入、编码校验、预览确认、提交给班长、处理退回修改 |
| 分析班长 | 审核分析员提交的数据，可通过或退回修改 |
| 分析室主任 | 审核班长通过的数据，下发车间，接收返回复核，做最终确认 |
| 车间 | 接收主任下发数据，下发给车队，接收车队返回结果 |
| 车队 | 接收车间下发数据，下发给指导组，接收指导组确认或反馈 |
| 指导组 | 确认无误或反馈不属实，反馈时必须填写原因和说明，可上传证据附件 |
| 系统管理员 | 管理用户、角色、部门、菜单、权限、字典、基础数据和系统配置，不等于可随意修改业务历史数据 |

## 4. 状态

| 状态 | 含义 | 可处理角色 |
| --- | --- | --- |
| 草稿 | 分析员保存但未提交 | 分析员 |
| 分析员已提交 | 分析员确认预览结果后提交 | 系统流转 |
| 班长待审核 | 等待分析班长审核 | 分析班长 |
| 主任待审核 | 班长审核通过，等待分析室主任审核 | 分析室主任 |
| 主任已下发车间 | 主任审核并下发到车间 | 系统流转 |
| 车间待确认 | 等待车间接收并下发车队 | 车间 |
| 车队待确认 | 等待车队接收并下发指导组 | 车队 |
| 指导组待确认 | 等待指导组确认无误或反馈不属实 | 指导组 |
| 指导组确认无误 | 指导组确认记录属实 | 系统流转 |
| 指导组反馈不属实 | 指导组反馈记录不属实 | 系统流转 |
| 返回主任待复核 | 逐级返回后等待分析室复核和主任处理 | 分析员、分析班长、分析室主任 |
| 主任最终确认 | 主任完成最终裁定 | 分析室主任 |
| 已入结果库 | 已进入 LKJ / 音视频违标结果数据库 | 只读为主 |
| 已撤销不计入 | 主任最终决定撤销，不计入结果统计 | 只读为主 |

状态规则：

1. 草稿可以修改和删除草稿内容，但删除草稿也应保留必要操作记录。
2. 已提交后不得直接编辑关键字段，必须通过退回修改或授权修改流程。
3. 指导组反馈不属实后，记录不得被物理删除。
4. 已入结果库的数据不得直接覆盖历史，只能形成新版本或追加更正记录。
5. 已撤销不计入的数据保留历史、附件和日志，但不进入统计计数。

| 机车 | 是 | 机车型号或机车号 |
| 车次 | 是 | 关联违章记录的车次 |

## 6. 录入规则

1. 日期必须分为提报日期和违章发生日期。
2. Excel 导出标题/副标题包含提报日期。
3. Excel 表格列只导出违章发生日期，不单独导出提报日期列。
4. 工号输入后自动带出责任人。
5. 违标编码必须来自数据库。
6. 选择违标编码后，系统自动带出性质、类别、类型。
7. 违标编码不存在时不允许提交。
8. 编码、性质、类别、类型不一致时，必须阻止提交；如业务确需覆盖，必须走授权覆盖并记录日志。
9. 录入后必须先进入预览，再确定提交。
10. 预览表格应接近 Excel 样式，便于业务人员核对。
11. 预览表格支持勾选 1-N 条记录提交。
12. 未勾选记录仍可留在预览或草稿批次中，不得误提交。
13. 预览页必须展示校验状态、错误原因、疑似重复提示。

授权覆盖规则：

- 只能由具备授权覆盖权限的角色操作。
- 必须填写覆盖原因。
- 必须记录覆盖前后的编码、性质、类别、类型。
- 覆盖后仍需保留原始编码快照和字段变更日志。

## 7. Excel 导入规则

Excel 导入是业务证据链的一部分，不允许直接把导入数据写入正式业务表。

导入规则：

1. 保存原始 Excel 文件。
2. 保存每一行原始 JSON 快照。
3. 导入数据先进入 staging / 预览表。
4. 系统按行校验，并分为校验通过、需要确认、校验失败。
5. 校验通过的数据可以勾选提交。
6. 需要确认的数据必须展示原因，由有权限人员确认后才能提交。
7. 校验失败的数据不得提交。
8. 支持下载错误报告。
9. 疑似重复要提示，不自动删除。
10. 导入批次必须记录导入人、导入时间、文件名、模板类型、总行数、成功行数、需要确认行数、失败行数。

预览分区：

| 分区 | 含义 | 是否可提交 |
| --- | --- | --- |
| 校验通过 | 编码、人员、部门、日期、内容等均通过校验 | 可提交 |
| 需要确认 | 存在疑似重复、字段轻微不一致、需要授权确认的情况 | 确认后可提交 |
| 校验失败 | 存在硬性错误 | 不可提交 |

## 8. 常见校验错误

| 错误 | 处理规则 |
| --- | --- |
| 违标编码不存在 | 阻止提交，提示维护编码字典或修正编码 |
| 编码与性质不匹配 | 阻止提交；授权覆盖时记录原因和字段变更 |
| 工号不存在 | 阻止提交，提示维护人员基础数据 |
| 工号与姓名不一致 | 标记为需要确认或阻止提交，按权限策略处理 |
| 责任部门不存在 | 阻止提交，提示维护组织基础数据 |
| 违章日期为空 | 阻止提交 |
| 拟考核内容为空 | 阻止提交 |
| 疑似重复导入 | 标记为需要确认，不自动删除 |

疑似重复判断建议：

- 违章发生日期相同。
- 时间相近。
- 工号或责任人相同。
- 机车、车次、地点相近。
- 违标编码相同或相近。

疑似重复只做提示，不自动合并、不自动删除。

## 9. 权限原则

权限不能只靠前端按钮隐藏，必须由后端接口再次校验。

权限分层：

| 权限类型 | 含义 | 示例 |
| --- | --- | --- |
| 功能权限 | 能否进入菜单或页面 | 进入每日 LKJ 音视频违标公示页面 |
| 操作权限 | 能否执行具体动作 | 新增、导入、预览确认、提交、审核、下发、反馈、复核、最终确认、导出 |
| 数据权限 | 能处理哪些组织、人员、批次和流程记录 | 车间只能看下发到本车间的数据 |
| 字段权限 | 能否编辑关键字段 | 授权覆盖编码、性质、类别、类型 |

导出权限：

1. 导出权限必须独立于查看权限。
2. 能看不等于能导出。
3. 所有导出必须由后端校验权限。
4. 所有导出必须写导出日志。
5. 导出日志必须记录导出人、导出时间、导出条件、导出数量、文件名、导出页面。

后端校验必须包含：

- 当前用户是否登录。
- 当前用户是否有功能权限。
- 当前用户是否有当前操作权限。
- 当前用户是否有当前记录的数据权限。
- 当前记录是否处于允许操作的流程状态。
- 当前用户是否具备字段修改、附件查看、导出、最终入库等特殊权限。

## 10. 信箱、待办、聊天协同关系

| 能力 | 定位 | 规则 |
| --- | --- | --- |
| 流程 | 改变业务状态 | 只有流程动作能改变正式业务状态 |
| 待办 | 当前必须处理的任务 | 只显示当前用户必须处理的任务 |
| 信箱 | 正式业务流转消息 | 用于接收、查看、归档正式业务流转消息 |
| 聊天 | 非正式协同提醒 | 用于沟通提醒，不代表业务已授权 |

关系规则：

1. 流程改变业务状态。
2. 待办只显示当前必须处理的任务。
3. 信箱是正式业务流转消息。
4. 聊天是非正式协同提醒。
5. 聊天业务卡片不代表授权。
6. 点击业务卡片必须重新做后端权限、数据权限、流程状态校验。
7. 聊天中分享业务卡片、文件、附件下载都必须记录日志。
8. 信箱消息不能替代待办处理，待办完成必须来自正式流程动作。

## 11. 不属实反馈

指导组可以反馈不属实，但反馈不属实不能直接删除原违章记录。

反馈必填内容：

| 内容 | 规则 |
| --- | --- |
| 原因类型 | 必须选择 |
| 说明 | 必须填写 |
| 证据附件 | 支持上传 |

原因类型建议：

- 人员不属实。
- 部门不属实。
- 时间不属实。
- 地点不属实。
- 车次或机车不属实。
- 违标编码不属实。
- 拟考核内容不属实。
- 其他。

主任最终处理结果：

| 处理结果 | 含义 |
| --- | --- |
| 维持 | 认为反馈不影响原认定，继续进入最终确认 |
| 退回复核 | 要求分析室或前序节点重新核对 |
| 撤销不计入 | 确认反馈成立，记录保留但不计入结果库统计 |

不属实反馈规则：

1. 必须写业务日志。
2. 必须保留反馈前记录快照。
3. 必须保留反馈原因、说明和附件。
4. 不能物理删除原违章记录。
5. 不能绕过逐级返回和分析室复核。

## 13. 结果库规则

结果库是最终业务结果，不允许把未完成流程、未校验、缺少日志证据链的数据写入结果库。

结果库规则：

1. 入库数据必须有完整流程日志。
2. 入库时保存记录快照。
3. 入库快照应包含人员快照、组织快照、违标编码快照、性质、类别、类型、拟考核内容、附件引用、流程摘要。
4. 后续修改不能直接覆盖历史。
5. 结果库支持版本。
6. 被撤销的记录保留历史，但标记不计入。
7. 入库后原则上只读；如需更正，必须走更正流程并形成新版本。
8. 结果库导出必须校验独立导出权限并写导出日志。

版本规则：

| 场景 | 处理 |
| --- | --- |
| 初次入库 | 生成版本 1 |
| 入库后更正 | 生成新版本，旧版本保留 |
| 撤销不计入 | 保留历史版本，标记不计入 |
| 重新维持或恢复计入 | 生成新版本并记录原因 |

## 14. 第一阶段 MVP

第一阶段只要求跑通以下闭环：

```text
录入/导入
→ 校验
→ 预览
→ 提交
→ 审核
→ 下发
→ 反馈/确认
→ 复核
→ 最终确认
→ 入结果库
```

第一阶段必须做：

- 手工录入。
- Excel 导入。
- 违标编码校验。
- 预览确认。
- 1-N 条勾选提交。
- 班长审核。
- 主任审核。
- 主任下发车间。
- 车间下发车队。
- 车队下发指导组。
- 指导组确认无误。
- 指导组反馈不属实。
- 逐级返回。
- 分析室复核。
- 主任最终确认。
- 入结果库。
- Excel 导出。
- 附件上传和查看。
- 信箱消息。
- 待办任务。
- 业务证据链日志。
- 后端权限、数据权限、流程状态校验。

第一阶段暂不做：

- 完整 IM。
- 完整 NAS。
- 复杂工作流引擎。
- 统计大屏。
- 多租户。
- 多数据源。
- 分布式任务。
- 大规模监控体系。

## 15. 后续开发任务拆分建议

建议后续按纵向切片拆分开发任务：

1. 基础数据：人员、工号、责任部门、车间、车队、指导组、违标编码字典。
2. 业务主表：每日 LKJ 音视频违标公示记录、批次、状态、快照字段。
3. 手工录入：录入表单、工号带出责任人、编码带出性质/类别/类型。
4. Excel 导入：原始文件保存、行级 JSON 快照、staging / 预览表、错误报告。
5. 校验服务：编码、人员、部门、日期、拟考核内容、疑似重复校验。
6. 预览提交：Excel 样式预览、校验分区、1-N 条勾选提交。
7. 审核流转：班长审核、主任审核、主任下发车间。
8. 下发确认：车间下发车队、车队下发指导组、指导组确认无误。
9. 不属实反馈：原因类型、说明、证据附件、逐级返回、分析室复核。
10. 主任最终确认：维持、退回复核、撤销不计入、入结果库。
11. 结果库：入库快照、版本、只读查询、撤销不计入标记。
12. 权限：功能权限、操作权限、数据权限、字段权限、独立导出权限。
13. 日志证据链：业务动作日志、字段变更日志、附件日志、导出日志、分享日志。
14. 信箱和待办：流转消息、当前处理任务、已处理记录。
15. 聊天协同：业务卡片、非正式提醒、重新鉴权、聊天卡片发送日志。
16. 前端页面：列表页、录入页、导入预览页、详情抽屉、审核页、结果库页。
17. 导出模板：标题/副标题含提报日期，表格列只导出违章发生日期。
18. 安全审查：越权访问、跨部门数据、导出权限、附件下载、流程状态绕过。
