# 每日 LKJ 音视频违标公示状态流转矩阵

本文定义“违章管理 > 每日 LKJ 音视频违标公示”的第一阶段状态流转矩阵。该矩阵用于后续后端状态机、权限校验、待办、信箱和业务日志实现。

本业务是“违章管理”模块下的样板业务，不是整个平台本身。第一阶段只实现必要主流程和不属实反馈闭环，不引入复杂会签、并行审批或通用工作流引擎。

## 1. 状态枚举表

| 状态 | 建议编码 | 含义 | 处理角色 |
| --- | --- | --- | --- |
| 草稿 | `DRAFT` | 分析员录入或导入后暂存，尚未提交 | 分析员 |
| 分析员已提交 | `ANALYST_SUBMITTED` | 分析员已确认预览并提交 | 系统流转 / 分析班长 |
| 班长待审核 | `LEADER_PENDING` | 等待分析班长审核 | 分析班长 |
| 主任待审核 | `DIRECTOR_PENDING` | 班长审核通过，等待分析室主任审核 | 分析室主任 |
| 主任待下发车间 | `DIRECTOR_APPROVED_PENDING_DISPATCH` | 主任审核通过后，等待主任下发责任车间 | 分析室主任 |
| 主任已下发车间 | `DIRECTOR_DISPATCHED_WORKSHOP` | 主任审核通过并下发到车间 | 车间 |
| 车间待确认 | `WORKSHOP_PENDING` | 等待车间接收并下发车队 | 车间 |
| 车队待确认 | `TEAM_PENDING` | 等待车队接收并下发指导组 | 车队 |
| 指导组待确认 | `GUIDE_PENDING` | 等待指导组确认无误或反馈不属实 | 指导组 |
| 指导组确认无误 | `GUIDE_CONFIRMED` | 指导组确认记录属实 | 系统流转 / 车队 / 车间 / 分析室 |
| 指导组反馈不属实 | `GUIDE_REJECTED` | 指导组反馈记录不属实 | 系统流转 / 车队 / 车间 / 分析室 |
| 返回主任待复核 | `RETURNED_DIRECTOR_RECHECK` | 逐级返回后等待分析室复核和主任处理 | 分析员、分析班长、分析室主任 |
| 主任最终确认 | `FINAL_CONFIRMED` | 主任完成最终裁定，等待入库或撤销 | 分析室主任 |
| 已入结果库 | `ARCHIVED` | 已进入 LKJ / 音视频违标结果数据库 | 只读为主 |
| 已撤销不计入 | `CANCELLED_EXCLUDED` | 主任最终决定撤销，保留历史但不计入统计 | 只读为主 |

实现说明：

1. `分析员已提交` 可作为提交动作后的业务状态；后端可同步生成班长待办，并在待办视图中表现为“班长待审核”。
2. 如果实现上希望减少中间状态，可将 `分析员已提交` 和 `班长待审核` 合并为同一数据库状态，但对外展示、日志和文档仍必须能区分“分析员已提交”和“班长待审核”的业务语义。
3. `已入结果库` 和 `已撤销不计入` 均为终态；后续更正必须走版本化更正流程，不允许直接改原记录。

实现补充：

1. `主任审核通过` 不应设计为 `主任待审核 -> 主任待审核` 自循环，正式流转为 `主任待审核 -> 主任待下发车间`。
2. `班长待审核` 可作为待办视图状态，也可作为数据库状态。若数据库使用 `分析员已提交` 表示该阶段，后端仍必须按班长待审核规则校验角色和数据权限。
3. `车间待确认` 可作为车间接收后的显式状态。若第一阶段不做“接收”按钮，则 `主任已下发车间` 可直接由 `车间下发车队` 处理，但页面展示应能表达“车间待处理”。
4. 所有“退回”“反馈不属实”“撤销不计入”动作必须填写原因；没有原因不得提交。
5. 附件不是状态流转的替代条件。附件上传必须绑定业务记录、上传人、上传时间和业务动作。

## 4. 禁止流转规则

1. `已入结果库` 后不得直接修改原业务记录、编码快照、人员快照、组织快照和结果快照。
2. `已入结果库` 后如需更正，只能走后续版本化更正流程，生成新版本，旧版本保留。
3. `已撤销不计入` 后不得物理删除记录、附件和日志。
4. `指导组反馈不属实` 不能直接删除原记录，必须逐级返回并由分析室复核、主任最终裁定。
5. `聊天协同`、聊天业务卡片、普通通知、文件分享不能改变正式流程状态。
6. `信箱消息` 不能替代待办处理；业务状态只能由正式流程动作改变。
7. `待办完成` 不能脱离业务动作单独完成，必须由审核、下发、反馈、复核、最终确认等动作驱动。
8. 未通过编码校验、人员校验、部门校验或必填字段校验的记录不得提交班长。
9. 校验失败的 Excel 导入行不得通过批量勾选绕过提交。
10. 前端隐藏按钮不构成权限控制，后端不得信任前端传入的可操作状态。
11. 非当前责任车间、车队、指导组的数据不得被跨组织处理。
12. 系统管理员拥有系统配置权限，不等于可以绕过业务流程修改历史业务数据。
13. 导出、附件下载、附件查看不得因为有查看权限就自动放行，必须校验独立权限。
14. 不允许通过直接更新数据库字段绕过业务日志、字段变更日志和流程状态校验。

## 5. 后端校验原则

每个流程动作必须在后端同时校验以下内容：

| 校验项 | 要求 |
| --- | --- |
| 登录态 | 当前用户必须已登录，登录态未过期 |
| 功能权限 | 当前用户必须有进入每日 LKJ 音视频违标公示功能的权限 |
| 操作权限 | 当前用户必须有当前动作对应的权限码 |
| 数据权限 | 当前用户只能处理自己组织、角色、任务范围内的数据 |
| 当前状态 | 记录当前状态必须与矩阵中的允许状态一致 |
| 目标状态 | 目标状态必须是矩阵允许的目标状态，不能由前端自由传入 |
| 目标对象范围 | 下发车间、车队、指导组必须在合法组织层级关系内 |
| 任务归属 | 当前用户必须是待办接收人、接收部门成员或接收角色成员 |
| 必填业务参数 | 退回、反馈不属实、撤销不计入等动作必须有原因和说明 |
| 字段权限 | 关键字段修改、授权覆盖、最终确认、入库必须校验特殊权限 |
| 附件权限 | 上传、查看、下载附件必须校验业务记录权限和附件权限 |
| 重复提交 | 同一记录同一动作应防止重复提交和并发覆盖 |
| 日志写入 | 状态改变前后必须写业务证据链日志和必要字段变更日志 |

后端实现要求：

1. 目标状态由后端根据当前状态和动作计算，不允许前端直接提交目标状态。
2. 每个动作应在同一事务内完成状态更新、待办更新、信箱消息、业务日志和字段变更日志。
3. 状态更新必须使用乐观锁、版本号或状态条件更新，避免并发重复流转。
4. 数据权限必须基于后端登录用户、角色、部门和业务归属计算，不能只依赖前端查询条件。
5. 待办和信箱是流程动作的结果，不是流程状态的来源。

## 6. MVP 范围说明

第一阶段只实现必要主流程和不属实反馈闭环：

```text
录入/导入
→ 校验
→ 预览
→ 提交班长
→ 班长审核
→ 主任审核
→ 下发车间
→ 车间下发车队
→ 车队下发指导组
→ 指导组确认无误 / 指导组反馈不属实
→ 返回复核
→ 主任最终确认
→ 入结果库 / 撤销不计入
```

第一阶段必须覆盖：

- 状态机校验。
- 角色和操作权限校验。
- 数据权限校验。
- 待办任务生成和完成。
- 信箱流转消息。
- 业务证据链日志。
- 不属实反馈原因、说明和证据附件。
- 结果库入库快照。

第一阶段暂不做：

- 复杂会签。
- 并行审批。
- 可视化流程设计器。
- 复杂工作流引擎。
- 完整 IM 流程驱动。
- 多租户流程隔离。
- 跨系统自动流转。

## 7. 后续需要确认的问题

1. `分析员已提交` 和 `班长待审核` 是否都作为数据库状态保存，还是一个作为数据库状态、一个作为待办展示状态。
2. 车间是否需要单独“接收确认”动作；如果需要，应补充“车间接收”动作和 `车间待确认` 的入口规则。
3. 是否允许主任审核通过后自动下发车间，还是必须保留主任待下发车间状态。
4. 指导组确认无误后，是否必须逐级经过车队、车间手动返回，还是系统自动返回分析室。
5. 不属实反馈被主任判定“维持”后，是否直接允许入结果库，还是必须再次下发确认。
6. 主任退回时，退回目标是分析班长、分析员，还是由主任选择具体复核节点。
7. 授权覆盖编码、性质、类别、类型的角色范围需要进一步明确。
